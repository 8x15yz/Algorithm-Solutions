r, e, v 문자 뒤에 오는 숫자들을 모두 더해서 형식에 맞게 재구성하는 작업
조건으로:
1. rev 뒤에는 2개까지 올 수 있음 (ex. r10)
2. 뒤에 2개까지 오지만, 11 이상부터는 무효. 11은 그냥 1으로 수집 

def solution(data):
    res = 0
    for a in range(len(data)):
        if data[a] in 'rev':
            if data[a+1:a+2] in 'rev':
                continue
            if data[a+1:a+3].isdigit() and 10 >= int(data[a+1:a+3]):
                res += int(data[a+1:a+3])
            else:
                res += int(data[a+1:a+2])
    res = str(res)[0] + '월 ' + str(res)[1:] + '일'
    return res

04 코드 품질

⚠️ 문제 있음 — 3가지

① 변수명: a → i, res → total
② data[a] in 'rev': 빈 문자열도 포함되어 오해 유발 → data[i] in {'r','e','v'}
③ 슬라이스 중복: one, two 변수로 분리 

def solution(data):
    total = 0
    for i in range(len(data)):
        if data[i] not in {'r', 'e', 'v'}:
            continue
        one = data[i+1:i+2]
        if not one.isdigit():
            continue
        two = data[i+1:i+3]
        if two.isdigit() and int(two) <= 10:
            total += int(two)
        else:
            total += int(one)
    s = str(total)
    return s[0] + '월 ' + s[1:] + '일'

💬 04.24

"" in "rev" 가 True인 이유

파이썬 공식 스펙에 명시된 동작이에요.

파이썬의 in 연산자 — 문자열 버전 정의

x in s → s 안에 x가 부분문자열(substring)로 존재하면 True

빈 문자열 ""은 모든 문자열의 부분문자열이에요.
수학적으로, 빈 집합은 모든 집합의 부분집합인 것과 같은 논리예요.

"" in "rev"    # True
"" in "hello"  # True
"" in ""       # True  ← 빈 문자열 안에도 True

왜 이게 문제가 됐냐면

data[a+1:a+2]  # a가 마지막 인덱스일 때 → ""
"" in 'rev'    # True ← 숫자 체크 전에 통과해버림

원래 의도는 "다음 문자가 r/e/v면 건너뛰어라" 였는데,
a가 문자열 끝일 때 슬라이스가 ""를 반환하고,
"" in 'rev'가 True라서 continue가 실행되어 버려요.

이 경우엔 결과적으로 건너뛰니까 크래시는 안 나지만,
의도와 다른 이유로 통과하는 거라 코드 읽는 사람이 헷갈려요.

리스트/집합 in과 비교

"" in ["r", "e", "v"]   # False ← "" 원소가 없으니까
"" in {"r", "e", "v"}   # False

"" in "rev"             # True  ← 부분문자열 검색이라서

문자열 in만 substring 검색이고, 리스트/집합 in은 원소 검색이에요.
그래서 data[i] in {'r','e','v'}처럼 집합으로 쓰면 이 혼란이 없어요.

def solution(data):

    res = []

        sum_skill = sum(arr[1:])

01 정확성

⚠️ 정렬 버그·역순 정렬 누락·문자 분리 버그 3가지 발견 (제공 예시는 우연히 통과)

① 정렬 버그: res[0](현재 최고점)과만 비교하므로 최고점보다 낮은 원소는 모두 맨 뒤에 추가됨. 예) [A=100, B=46, C=90] 입력 시 res = [('A',100),('B',46),('C',90)]으로 C가 B 뒤에 위치. 제공 예시는 최고점 A가 첫 번째로 입력되어 우연히 통과하지만 순서가 바뀌면 오답이 됨.
② 알파벳 역순 정렬 누락: 결과를 알파벳 역순으로 출력해야 하지만 f_res에 대한 정렬 로직이 없음. 예시가 선발자 1명이라 드러나지 않음.
③ f_res += fin[0] 문자 분리: fin[0]이 문자열이므로 +=가 각 문자를 개별 원소로 추가. 예시 이름이 모두 한 글자라 통과하지만 여러 글자 이름이면 ['A','l','i','c','e']로 분리됨. f_res.append(fin[0])이 올바름.
④ 미사용 변수: log = "" 삭제 필요. 

def solution(data):
    scored = sorted(
        [(arr[0], sum(arr[1:])) for arr in data],
        key=lambda x: x[1],
        reverse=True
    )

    n = len(scored)
    limit = int(n * 0.3)

    if limit == 0:
        return []

    # ① limit으로 자른다 — 경계 마지막 사람의 점수를 기준점으로 저장
    cutoff = scored[limit - 1][1]

    # ② 경계(scored[limit-1])와 바로 다음(scored[limit])이 동점이면
    #    그 점수를 경계 안에서도 전원 제거
    if limit < n and scored[limit][1] == cutoff:
        selected = [name for name, score in scored[:limit] if score > cutoff]
    else:
        selected = [name for name, _ in scored[:limit]]

    return sorted(selected, reverse=True)

02 시간복잡도

⚠️ O(n²) — sorted() 활용으로 O(n log n)으로 개선 가능

루프 내 not in res 선형 탐색 O(n), res = [x] + res 앞 삽입 O(n) → 전체 O(n²). n ≤ 10이라 체감 차이는 없지만 sorted() 한 번으로 O(n log n)에 완전 정렬 가능.

03 공간복잡도

✅ 이상 없음

res, final, f_res 모두 O(n) 수준으로 적절.

05 보안

✅ 이상 없음

알고리즘 문제 특성상 외부 입력 위협 없음. 제약 조건(n ≤ 10)이 문제 단계에서 보장됨.

총평: sorted() 한 줄로 해결할 수 있는 정렬을 직접 구현하면서 정렬 버그가 발생했고, 역순 정렬과 append 사용도 누락됨. 동점자 처리 아이디어는 올바른 방향이었으나 잘못된 정렬 위에서 동작하므로 신뢰할 수 없음. sorted()로 재작성하면 코드가 절반으로 줄고 버그 세 가지가 동시에 해소됨.
